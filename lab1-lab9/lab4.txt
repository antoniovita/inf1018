3) Primeiro deslocamos a palavra word para a direita em 8 * bytenum bits, trazendo o byte desejado para a posição menos significativa. Em seguida, deslocamos esse valor 24 bits para a esquerda, colocando o bit de sinal do byte na posição mais significativa de um inteiro de 32 bits. Por fim, deslocamos 24 bits para a direita novamente. Como esse deslocamento é aritmético, por se tratar do tipo int, o bit de sinal é replicado, realizando a extensão de sinal e produzindo o valor correto do byte como inteiro com sinal.
int xbyte(packed_t word, int bytenum) {
    int byte = (int) word >> (bytenum * 8) & 0xff;
    return (byte << 24) >> 24;
}

4) Os resultados mudam porque o C interpreta e converte os valores de forma diferente dependendo de os tipos serem com sinal (signed) ou sem sinal (unsigned), e isso afeta diretamente a comparação. No Programa 1, como tudo envolve int sem especificar unsigned, os valores são tratados como signed int. Assim, o padrão de bits 0xffffffff corresponde a -1 em complemento a 2 e, como -1 é menor que 2, a expressão 0xffffffff < 2 resulta verdadeira. No Programa 2, como as variáveis são unsigned int, não existe bit de sinal e os valores são sempre interpretados como não negativos. Dessa forma, 0xffffffff passa a representar 4294967295 (em 32 bits), que é maior do que 2, e por isso a comparação indica que 0xffffffff > 2. Já no Programa 3, embora x seja int e seja impresso como -1, a comparação envolve um unsigned int (y), e pelas conversões usuais do C o operando com sinal é convertido para unsigned antes da comparação; assim, -1 é convertido para 4294967295 (em 32 bits), fazendo a expressão equivalente a 4294967295 < 2, que é falsa. Além disso, nos Programas 2 e 3 a saída pode causar confusão porque foi usado %d para imprimir uma variável unsigned int, quando o correto seria %u.

5) Valor em signed char (8 bits)
sc = -1 | em 8 bits, -1 é representado com todos os bits 1:
sc = 1111 1111

Quando ele passa pela conversão de tipo, primeiro há o aumento de tamanho para 32 bits (como sc tem sinal, faz extensão de sinal)
extensão de sinal copia o bit mais significativo, que aqui é 1, para os bits novos:
11111111  ->  11111111 11111111 11111111 11111111  (0xFFFFFFFF)

Interpretação como unsigned int (32 bits)
Os bits não mudam, só muda a interpretação:
11111111 11111111 11111111 11111111 = 2^32 - 1 = 4294967295

Então ui fica com representação interna 0xFFFFFFFF.

Código para teste:

#include <stdio.h>
void dump_ui(unsigned int x) {
  unsigned char *p = (unsigned char*)&x;
  size_t i = 0;

  printf("ui como unsigned: %u\n", x);
  printf("ui em hex: 0x%08X\n", x);
  printf("bytes: ");
  while (i < sizeof(x)) {
    printf("%02X ", p[i]);
    i++;
  }
  printf("\n");
}

int main(void) {
  signed char sc = -1;
  unsigned int ui = sc;

  printf("sc como int: %d\n", (int)sc);
  dump_ui(ui);
  return 0;
}
